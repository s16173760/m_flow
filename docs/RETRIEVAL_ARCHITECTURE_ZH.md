# M-Flow 核心检索架构解析

## 平坦检索的根本问题

每个 RAG 系统都面临同一个问题：给定用户查询，哪些存储的知识是相关的？

主流做法是将文档切块、向量化、存入向量库，检索时按余弦相似度排序。这本质上只回答一个层级的问题："哪段文本和查询语义最接近？" 对简单的事实查找有效，但在以下场景中崩溃：

- **答案跨越多个文档**——切块之间没有结构性连接，系统无法将分散在不同文档中的关联信息组合起来
- **用户提问的粒度不匹配**——宏观问题检索到过于琐碎的片段，微观问题检索到过于笼统的摘要
- **两份文档讨论同一个实体但语境不同**——向量空间中它们可能距离很远，系统无法桥接

核心问题是：平坦向量检索**丢弃了结构**。它知道某段文本和查询"接近"，但完全不知道这段文本*处于知识拓扑的什么位置*。

## M-Flow 的方案：图路由 Bundle Search

M-Flow 用**图路由检索**替代平坦检索，在分层知识拓扑上操作。核心洞察是：

> 不要只找到"什么匹配了"——找到匹配点所属的*知识结构*，然后对整个结构评分。

### 知识拓扑：倒锥结构

M-Flow 将所有摄入的知识组织为一个四层有向图，形成一个**倒锥（inverted cone）**：

| 层级 | 角色 | 在锥中的位置 |
|------|------|-------------|
| **FacetPoint** | 原子断言——一条精确的事实或数据点 | **锥尖**：语义最尖锐，匹配最精确 |
| **Entity** | 命名事物——横穿所有层级的桥梁 | **锥尖**：语义高度聚焦于一个名称 |
| **Facet** | 截面维度——语义焦点的一个角度或方面 | **中间层** |
| **Episode** | 有边界的语义焦点 | **锥底**：语义最丰富，提供最终的召回落点 |

这个结构的方向性是**反直觉的**：在传统的知识图谱或分类树中，越往下越具体。但在 M-Flow 中，搜索的**入口在锥尖**（细粒度的 Entity 和 FacetPoint 是最容易被向量搜索精确命中的），而搜索的**目标在锥底**（Episode 是最终返回给用户的知识单元）。信息流从尖锐的匹配点向下汇聚到宽广的语义落点。

这打破了"从上到下浏览"的传统检索范式。用户不是在层级中逐层缩小范围，而是系统在最尖锐的点上捕获信号，然后**沿图结构向下传播**到它所归属的完整语义单元。

### 第一个打破常规的设计：边也有语义

在几乎所有知识图谱系统中，边只是类型标签（`works_at`、`located_in`）。查询一个图时，你要么遍历边，要么忽略边——边本身不携带可被搜索的语义。

M-Flow 打破了这一点：每条边都携带自然语言描述文本，这些文本同样被向量化、同样参与搜索。

这意味着边不再是被动的连接器，而是**主动的语义过滤器**。在代价传播阶段，系统不仅知道"两个节点之间存在连接"，还知道"这条连接关系本身与当前查询有多相关"。一条语义上无关的边会贡献高代价，有效阻断不相关的路径——即使两端的节点都被向量搜索命中了。

### Bundle Search 的工作方式

当查询到达时，系统不是简单地找到最近的节点。它通过评估图中所有可能到达每个 Episode 的路径，找到**最优的 Episode**。

**阶段一 —— 在锥尖广撒网**

查询被向量化后，同时在七个向量集合中搜索——从锥尖到锥底覆盖每一层。每个集合返回最多 100 个候选。

最容易被精确命中的是锥尖处的节点：一个 Entity 名称、一个 FacetPoint 的断言。这些细粒度锚点的语义极度聚焦，向量距离小。锥底的 Episode 摘要也可能被命中，但因为语义更宽泛，匹配通常不如锥尖精确。

**阶段二 —— 投影到图中**

这些锚点被用作进入知识图谱的入口节点。系统提取它们周围的子图——边、邻居、连接关系，然后扩展一跳邻居。这将一组孤立的向量命中点转化为一个连通的拓扑结构。

**阶段三 —— 从锥尖向锥底传播代价**

这是核心步骤，也是 Bundle Search 的本质：**在锥尖捕获信号，沿图边向锥底传播，在 Episode 处汇聚评分。**

对于子图中的每个 Episode，系统评估从锚点到达它的所有可能路径：

| 路径 | 从哪里出发 | 经过什么 |
|------|----------|---------|
| 直接命中 | Episode 自身 | 无（但施加惩罚，见下文） |
| Facet 路径 | Facet | → Episode |
| FacetPoint 路径 | FacetPoint | → Facet → Episode |
| Entity 直接路径 | Entity | → Episode |
| Entity-Facet 路径 | Entity | → Facet → Episode |

每条路径累积代价：
- **起始代价**——锚点的向量距离（信号的尖锐程度）
- **边代价**——沿途每条边的向量距离（连接关系与查询的相关度）加跳跃惩罚
- **未命中惩罚**——边没有被向量搜索命中时的默认高代价

Episode 的最终得分是**所有路径中的最小代价**。

**阶段四 —— 排序与组装**

Episode 按 bundle 代价排序，取 Top-K。输出内容由调用方指定的显示模式决定：summary 模式返回 Episode 摘要，detail 模式返回 Episode 下的 Facet 和 Entity，highly_related_summary 模式只返回 Episode 摘要中与命中 Facet 相关的段落。

### 第二个打破常规的设计：取最小值而非平均值

为什么取最小值？这是一个深思熟虑的检索哲学：**一条强的证据链就足以证明相关性。**

一个 Episode 可能有十个 Facet，其中九个与查询无关。传统做法可能平均所有路径的代价——被九条高代价路径拉高。Bundle Search 只看那条最好的路径。只要有一个 Facet 通过低代价路径连接到查询，这个 Episode 就应该被检索到。

这对应了人类记忆的工作方式：你想起一件事，通常是因为某一个线索足够强烈，而不是因为所有线索都指向它。

### 第三个打破常规的设计：惩罚直接命中

最反直觉的设计：当查询直接匹配了 Episode 摘要时，系统反而**对这条路径施加额外惩罚**。

为什么惩罚最直接的命中？因为 Episode 摘要是高层概括——它们和很多查询"看起来相关"。一个关于"项目管理"的 Episode 摘要可能和任何提到"项目"或"管理"的查询都有不错的向量距离。但这种匹配是**宽泛的、缺乏焦点的**，这也反映了众多 RAG 系统检索噪声的根本原因。

系统的设计偏好是：**如果存在从锥尖（FacetPoint、Entity）出发的更精确路径，即使多走几跳，也优先选择它。** 直接的 Episode 命中只在没有更好替代路径时才胜出。

这确保了检索结果的精确性——不是"什么都沾点边"的宽泛摘要，而是有具体证据链支撑的 Episode。

### 为什么有效：拓扑论证

根本优势在于：**图拓扑编码了向量本身无法捕获的知识组织结构**。

**多粒度自动路由。** "数据库迁移发生了什么？"——宏观问题直接匹配 Episode 摘要，虽然有直接命中惩罚，但因为没有更精确的锥尖路径，直接命中仍然胜出。"P99 目标是否低于 500ms？"——精确问题强匹配一个 FacetPoint，从锥尖出发经两跳到达 Episode，起始距离极小使总代价很低。系统不需要"选择"粒度——倒锥拓扑将每个查询路由到它自然匹配的层级。

**跨文档实体桥接。** 当"张博士在 MIT 工作"出现在文档 A、"MIT 发表了量子计算突破"出现在文档 B 时，两个 Episode 共享同一个 Entity 节点"MIT"。关于 MIT 的查询在锥尖命中该 Entity，路径代价同时向下传播到两个 Episode。用户从两个独立来源获得结果，无需 LLM 推理——图拓扑本身就是桥接。

**结构噪声过滤。** 在平坦检索中，语义相似但主题无关的切块排名靠前。在 Bundle Search 中，该切块必须沿边追溯到某个 Episode。如果沿途的边与查询无关（高边向量距离），路径代价膨胀，结果下沉。图是语义噪声的结构过滤器。

**代价传播即推理。** 图中的每条路径代表一条推理链："查询匹配了这个事实，该事实属于这个主题，这个主题是这个事件的一部分。"路径代价量化了推理链的紧密程度。系统通过代价算术在 2-3 跳内完成多跳推理，不需要在检索阶段调用 LLM。

### 自适应置信度

并非所有向量集合对每个查询都同样可靠。系统对每个集合计算两个信号：绝对匹配强度和区分度。然后系统将所有集合分为"节点类"和"边类"，按置信度比例分配权重。如果 Entity 集合的置信度远高于 Facet 集合，系统会加大 Entity 路径的影响力。

系统在每个查询上自适应调整检索策略——不是固定的权重，而是根据"这次搜索中哪个粒度的命中最可靠"动态决定。

### 补充：一个额外的调节机制

当一个 Facet 的向量距离极小（与查询高度吻合），系统会大幅降低经过它的路径上的边代价和跳跃代价。逻辑是：如果某个 Facet 几乎完全对应查询，那它到 Episode 的连接几乎确定是正确的，不需要边语义再去验证。

此外，系统中还包含查询预处理、并行多模式编排、输出裁剪等机制，此处不再展开。

## 总结

M-Flow 的检索不是"向量搜索加上一个图数据库"。图就是检索机制本身：

1. **倒锥拓扑**——细粒度锚点（Entity、FacetPoint）在锥尖捕获精确信号，宽广的 Episode 在锥底提供召回落点
2. **边作为语义过滤器**——不是被动连接，而是主动参与路径评分，阻断无关路径
3. **路径代价传播**——从锥尖到锥底，按最紧密的证据链对每个 Episode 评分
4. **取最小值**——一条强路径即足够，对应人类记忆的单线索触发
5. **惩罚直接命中**——偏好从精确锚点出发的路径，防止宽泛匹配霸占排名
6. **自适应置信度**——根据每次搜索中哪个锥层（粒度）的命中最可靠，动态调整策略

最终效果：同一个查询可以自然地找到宏观的 Episode 摘要、精确的原子事实、或跨文档的实体关联——取决于最低代价路径通往何处。信号在锥尖捕获，在锥底落地。
